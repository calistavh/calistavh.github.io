Type Conventional Commit

Sprint 1 Week 2 | 3 Maret 2024

Commit message adalah satu fitur dalam version control system yang tampaknya kecil namun sangat penting dalam software development. Konsepnya sederhana: kita memberi pesan singkat yang mendeskripsikan perubahan apa yang kita lakukan terhadap project. Pesan ini bermanfaat bagi para developer (termasuk diri sendiri) yang suatu saat ingin mencari version tertentu, misalnya untuk memperbaiki sebuah bug. Alih-alih mengulik kode semua version satu per satu, kita cukup melihat message dalam daftar commit untuk menemukan perubahan mana yang membuat bug tersebut muncul.

Jadi keberadaan commit message dapat membuat pekerjaan dalam tim maupun individu lebih efisien--jika message yang diberikan formatnya konsisten. Tapi nyatanya tidak ada aturan wajib dalam membuat commit message, sehingga saya sendiri pun terbiasa membuat message seadanya sejak awal belajar git. Baru dalam proyek PPL ini saya melihat ada yang sedikit berbeda dengan commit message teman-teman sekelompok dibandingkan punya saya.

feat: add <feature_name> feature

"Prefiks apa ini?" pikir saya. Saya lihat version history, ternyata semua commit message-nya diawali oleh prefiks yang serupa. Dipikir-pikir lagi, saya melihat gaya commit message seperti ini beberapa kali sebelumnya tapi belum pernah saya ulik sendiri. Jadi saya bertanya ke teman saya (Google) tentang maksud dari pelabelan commit message tersebut. Dari sana saya menemukan bahwa commit message juga memiliki konvensi yang dinamakan... ya, conventional commit.

Inti dari conventional commit, menurut website resminya, adalah struktur ini:

<type>[optional scope]: <description>
[optional body]
[optional footer(s)]

Kalau kita hapus semua bagian opsionalnya, maka didapat syntax paling sederhana <type>: <description>. Sesuai dengan commit message yang sering saya lihat. Website di atas menjelaskan dua type utama untuk commit, yaitu feat dan fix. Namun ada beberapa type lain yang dapat digunakan menurut cheat sheet ini. Berikut adalah beberapa type yang saya sudah atau ingin gunakan untuk project-project kedepannya.

feat

feat digunakan untuk memperkenalkan fitur baru--setidaknya menurut website, tapi saya ingin tahu apakah perubahan pada fitur dikategorikan ke dalam feat atau tidak. Dipikir lagi, definisi 'fitur' di sini tidak memiliki definisi yang jelas. Jadi untuk saat ini, saya mengambil kesimpulan bahwa jika terjadi perubahan pada fungsionalitas atau output program, maka perubahan tersebut dikategorikan sebagai feat.


fix

fix digunakan untuk memperbaiki bug yang muncul pada version sebelumnya, dengan deskripsi berisi bug tersebut. Saya sendiri belum pernah menggunakannya, tapi sepertinya type ini akan banyak muncul setelah banyak commit, testing, dan integrasi.

refactor

refactor digunakan untuk menandai perubahan kode yang tidak menambah fitur atau memperbaiki bug. Umumnya berhubungan dengan perbaikan code smell, implementasi design pattern, dan sebagainya. Saya pernah menggunakan refactor, tapi sepertinya salah type.


Seharusnya commit ini lebih cocok untuk feat karena fungsionalitas deaktivasi event berubah dari menunggu input user menjadi otomatis.

test

test digunakan untuk segala hal yang berkaitan dengan test, seperti membuat dan memperbaiki test. Cukup mudah dipahami karena type ini pasti digunakan secara eksklusif di file test.


build

build digunakan untuk segala hal yang berkaitan dengan build system seperti package, dependency, dan lain-lain. Sejauh ini saya menggunakan build dalam file seperti pubspec.


style

style digunakan untuk perubahan dalam file kode yang tidak mengubah makna kode itu sendiri, seperti penghapusan comment dan whitespace atau formatting. Saya juga belum pernah menggunakan type ini.

chore

chore digunakan untuk menandai perubahan miscellanous pada bagian-bagian project selain yang sudah disebutkan di atas. Misalnya, saya menandai perubahan nama file dan directory asset image sebagai chore.


Melabeli commit message dengan type tidak hanya membuat penelusuran commit lebih mudah bagi manusia, tetapi juga bagi mesin. Ini karena web conventional commit menyebutkan bahwa salah satu tujuan utama pembentukannya adalah untuk memfasilitasi pembuatan changelog secara otomatis. Jadi kesimpulannya, conventional commit membuat commit history menjadi lebih readable. Namun tentunya melabeli sebuah message dengan type tidak cukup tanpa disertai description yang singkat, padat, dan jelas.
