SOLID Principle untuk Object-Oriented Programming

Sprint 1 - 25 Februari 2024


Kini banyak bahasa pemrograman yang mendukung penggunaan paradigma object-oriented seperti Dart, yang akan saya dan kelompok saya gunakan untuk mengembangkan sebuah game mobile berbasis Flutter. Flutter banyak menggunakan objek class dalam bentuk widget dan component, sehingga penting untuk memahami good practice dalam object-oriented programming untuk memaksimalkan development proyek. Salah satu (atau lima) good practice tersebut adalah SOLID principle, yang terdiri atas lima prinsip berikut:

1. Single Responsibility Principle
Prinsip ini berarti sebuah class seharusnya memiliki satu tanggung jawab atau menjalankan satu tugas saja dibandingkan dengan class besar yang melakukan banyak hal. Single responsibility ini penting untuk membuat program lebih mudah dimengerti developer lain karena class-class menjadi pendek dan sederhana. Bugfixing dan testing pun menjadi mudah dengan setiap class berfokus pada tugas tertentu.

2. Open-Closed Priciple
Prinsip ini berarti sebuah class seharusnya terbuka untuk dilakukan ekstensi, tapi tertutup untuk dilakukan modifikasi. Ekstensi di sini berbentuk subclass, biasanya dengan method yang lebih spesifik, tapi method yang terdapat dalam superclass itu sendiri tidak berubah. Dalam konteks development skala besar, seorang developer tentunya tidak bisa sembarangan mengubah kode dalam class yang sudah dibuat developer lain hanya untuk menyesuaikan dengan kode miliknya.

3. Liskov Substitution Principle
Prinsip ini berarti substitusi subclass pada superclass, dalam kasus subclass harus dapat mengerjakan tugas yang dilakukan superclassnya. Sebelumnya pada open-closed principle, yang menjadi good practice adalah ekstensi yang berarti subclass tetap memiliki fungsi turunan dari superclass. Namun subclass tersebut bisa saja melanggar principle ini, misalnya jika dilakukan override pada method superclass yang mengubah return type.

4. Interface Segregation Principle
Prinsip ini berarti setiap class seharusnya hanya memiliki method yang merupakan bagian dari fungsionalitasnya. Maksudnya jika subclass diektensi dari superclass, pastikan bahwa method dalam superclass juga relevan bagi subclass. Oleh karena itu, class sebaiknya dibuat sekecil mungkin agar tidak berpotensi membuat subclass yang melanggar prinsip ini.

5. Dependency Inversion Principle
Prinsip ini berarti class sebaiknya bergantung pada abstraksi seperti abstract class atau interface dibandingkan implementasi atau concrete class. Prinsip ini bertujuan untuk mengurangi decoupling karena abstraksi lebih fleksibel, sehingga jika objek yang digunakan dalam dependency diganti, program tidak akan terpengaruh selama objek tersebut berasal dari abstraksi yang sama.

Melihat lima prinsip di atas, dapat disimpulkan bahwa untuk memanfaatkan object-oriented programming dengan baik, developer tidak hanya perlu memahami cara melakukan ekstensi pada class, tetapi juga cara mendesain class yang baik untuk diekstensi. Dengan demikian, SOLID principle penting untuk dipelajari kembali terutama pada awal-awal tahap pengembangan untuk menghindari refactoring yang berlebihan ke depannya.
